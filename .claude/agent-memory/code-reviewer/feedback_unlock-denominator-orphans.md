---
name: unlock-denominator-orphans
description: Progress/unlock percentage denominators can include content items that no drill/question path ever makes masterable, permanently capping the percentage below the unlock threshold
metadata:
  type: feedback
---

When a feature has (a) a mastery/progress percentage computed from "all drillable items tagged unit X" and (b) a separate drill/question builder that decides which of those items actually become answerable subjects, always check that every item counted in (a)'s denominator is reachable as a subject in (b) for that same unit/bucket.

**Why:** Found in M11 Read-Thai review (`lib/thai/queries.ts` `getUnitSummaries` vs `lib/thai/drill.ts` `buildSubjectPool`): 8 lesson-only "final sound" rows were seeded with `drillable:true` and tagged `unit:6`, so they counted toward unit 6's mastery denominator (38 items), but the drill-round builder never used them as subjects (only consonants-with-final and word-bank syllables). Max achievable mastery capped at 30/38 ≈ 79%, permanently below the 90% unlock threshold — silently blocking all downstream units forever. Nothing in lint/typecheck/build catches this because it's a cross-file logical mismatch, not a type error.

**How to apply:** When reviewing any "mastery %" / "unlock at N%" feature, trace the denominator's data source and the numerator's achievability path as two separate questions, and explicitly check they cover the same item set. Grep for the content `kind` values that appear in the denominator query and confirm each one is referenced as a subject somewhere in the drill/question builder — flag any that are lesson-only-but-marked-drillable. This is a distinct pattern from the already-known "streak cap verification" and "overdue forecast gap" memories — those are about time-window bounds, this is about denominator/numerator set mismatch.

**Round-2 correction — generalize the check, don't stop at one instance.** In the M11 re-review, the implementer fixed the exact 8 rows the round-1 finding named (`kind:"final"` items) but the *identical* pattern existed a second time in the *same unit*, undetected: 9 of 30 `kind:"syllable"` word-bank rows had `metadata.finalSound: null` and were therefore just as unreachable as drill subjects, capping the "fixed" unit at 70% instead of the prior 79% — still under the 90% threshold. I had verified the named instance in round 1 but did not generalize to "check every row/kind in this unit's denominator for reachability," so I missed it. **When re-reviewing a fix for this pattern, don't just re-check the named rows — write the reachability check as a query over the *entire* denominator set (e.g. `SELECT * FROM items WHERE unit=X AND drillable AND id NOT IN (<computed reachable-subject ids>)`) and confirm the result is empty, rather than spot-checking the specific rows the original finding called out.**

**Round-3 resolution — what a genuinely closed fix looks like.** After round 2 caught a second instance, the implementer's round-3 fix wasn't just another one-off flag flip: they factored the "is this reachable" filter into one pure, DB-free module (`lib/thai/reachability.ts`) consumed by *both* the request-time drill builder (`buildSubjectPool`) and a pre-write assertion in the seed script (`assertEveryDrillableItemIsReachable()`, called before any DB write, throws + exits non-zero on violation). This is the correct closure pattern for this bug class: a single shared implementation (not two call sites that can drift) plus a mechanical, always-run check (not "reviewer reads the code carefully each time"). When verifying a fix like this, confirm three things: (1) the consuming code actually imports the shared function rather than re-deriving the same logic inline elsewhere: (2) the check runs unconditionally before any write, not just in a test file that might not be run; (3) the check genuinely fires — don't just confirm it passes on current data, simulate a regression (clone the data, flip one flag, call the checker directly) and confirm it's caught. All three held here, confirmed via an independent re-implementation of the reachability logic against live DB data (matched exactly) plus an independent regression-simulation script using the real modules.
